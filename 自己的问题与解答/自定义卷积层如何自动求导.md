# Q:在我们创建的`Conv2D`自动求导时，有什么错误消息？(课本练习2)

**自定义 `Conv2D` 自动求导时的错误消息及原因**

**错误消息**（在尝试对 `Conv2D` 的输出进行反向传播时）：
```
RuntimeError: element 0 of tensors does not require grad and does not have a grad_fn
```
或更具体：
```
RuntimeError: grad can be implicitly created only for scalar outputs
```
（取决于调用方式）

**根本原因**  
自定义类中的 `forward` 调用了手写的 `corr2d` 函数，该函数使用 **Python 循环和索引赋值** 构造输出张量 `Y`：

```python
Y = torch.zeros((X.shape[0]-h+1, X.shape[1]-w+1))
for i in range(Y.shape[0]):
    for j in range(Y.shape[1]):
        Y[i, j] = (X[i:i+h, j:j+w] * K).sum()
```

这里的问题在于：
- `torch.zeros` 创建的 `Y` 默认 `requires_grad=False`，且没有 `grad_fn`。
- 循环内的 `Y[i, j] = ...` 是**原地（in-place）赋值**，这种操作会**破坏计算图**。每个 `Y[i,j]` 与右侧表达式之间的梯度链路没有被记录，因为赋值操作不是 PyTorch 的自动微分可追踪的操作。
- 因此，`Y` 与输入 `X` 和参数 `self.weight` 之间不存在任何可微的运算关系，`Y` 的 `grad_fn` 为 `None`。
- 当对损失（如 `(Y_hat - Y).sum()`）调用 `.backward()` 时，梯度无法从损失传播到 `self.weight` 和 `self.bias`，因为 `Y_hat` 没有连接到参数的计算图，从而引发上述错误。

**正确做法**：应使用 PyTorch 内置的向量化操作（如 `torch.nn.functional.conv2d` 或使用 `unfold` + 矩阵乘法）来构建计算图，避免索引赋值。


# Q:怎么让自定义的Conv2D能够自动求导？

要让自定义的 `Conv2D` 支持自动求导，**核心原则是：前向传播中的所有运算必须由 PyTorch 的张量操作（Tensor ops）构成，且不能使用原地（in-place）赋值**（如 `Y[i, j] = ...`），因为后者会切断计算图。

针对你代码中的情况，有三种可行的解决方案，按推荐程度排序如下：

### 方法一：直接使用 `torch.nn.functional.conv2d`（最推荐）
这是最标准、最高效的做法。`F.conv2d` 底层调用的是高性能的 CUDA 内核，并且已经内置了反向传播的公式，直接使用即可完美自动求导。

**注意**：你需要将权重 `self.weight` 的形状调整为 `(输出通道数, 输入通道数, 核高, 核宽)`，并确保输入 `x` 是四维张量 `(批量大小, 通道数, 高, 宽)`。

```python
import torch.nn.functional as F

class Conv2D(nn.Module):
    def __init__(self, kernel_size):
        super().__init__()
        # 假设输入和输出都是单通道（通道数为1），核形状为 (1, 1, h, w)
        if isinstance(kernel_size, int):
            h = w = kernel_size
        else:
            h, w = kernel_size
        # 权重形状: (输出通道=1, 输入通道=1, 高, 宽)
        self.weight = nn.Parameter(torch.randn(1, 1, h, w))
        self.bias = nn.Parameter(torch.zeros(1))

    def forward(self, x):
        # x 需要是 4D 张量 (batch, channels, height, width)
        # 如果传入的是 2D 灰度图，需要先 unsqueeze
        if x.dim() == 2:
            x = x.unsqueeze(0).unsqueeze(0)  # 变为 (1, 1, H, W)
        return F.conv2d(x, self.weight, self.bias)
```

---

### 方法二：用 `unfold` + 矩阵乘法重写 `corr2d`（纯 PyTorch 向量化）
如果你想保留手写互相关运算的感觉，同时避免使用循环和原地赋值，可以使用 `F.unfold` 将输入滑动窗口展开为矩阵，然后通过矩阵乘法一次性算出所有输出值。整个过程是纯张量乘法，PyTorch 会自动记录梯度。

```python
import torch.nn.functional as F

def corr2d_vectorized(X, K):
    """
    X: 2D 输入 (H, W) 或 4D 输入 (B, C, H, W)
    K: 2D 卷积核 (h, w)
    """
    # 统一处理为 4D 格式 (B, C, H, W)
    if X.dim() == 2:
        X = X.unsqueeze(0).unsqueeze(0)  # (1, 1, H, W)
    elif X.dim() == 3:
        X = X.unsqueeze(0)               # (1, C, H, W)
    
    B, C, H, W = X.shape
    h, w = K.shape
    
    # 将输入展开为 patches: (B, C*h*w, N)，N 为输出位置的个数
    patches = F.unfold(X, kernel_size=(h, w))  
    # 将 K 展平并调整形状用于矩阵乘法
    K_flat = K.view(-1, 1)  # (h*w, 1)
    
    # 矩阵乘法: (B, N, h*w) @ (h*w, 1) -> (B, N, 1)
    out = patches.transpose(1, 2) @ K_flat
    # 重塑为 (B, 1, H-h+1, W-w+1)
    out = out.transpose(1, 2).view(B, 1, H - h + 1, W - w + 1)
    
    return out.squeeze()  # 如果原来输入是 2D，这里去掉多余维度方便观察

# 在 Conv2D 中调用
class Conv2D(nn.Module):
    def forward(self, x):
        return corr2d_vectorized(x, self.weight) + self.bias
```

### 为什么原来的代码报错？
你原来的 `corr2d` 函数中：
```python
Y = torch.zeros((...))
Y[i, j] = (X[i:i+h, j:j+w] * K).sum()
```
`torch.zeros` 创建的新张量 `Y` 是没有 `grad_fn` 的“叶子节点”，而 `Y[i, j] = ...` 属于**原地索引赋值**。这种操作会就地修改 `Y` 的数据，但不会在 PyTorch 的计算图中记录“赋值”这一动作。因此，`Y` 和参数 `self.weight` 之间的数学关系链完全断裂，反向传播时梯度无法从损失传到权重，从而抛出 `RuntimeError`。

### 总结建议
- **工程实践**：直接用 **方法一**（`F.conv2d`），代码简洁、速度快、梯度准确。
- **学习理解**：用 **方法二**（`unfold` + 矩阵乘法），能帮你理解卷积的数学本质（Im2Col），且梯度流畅。
- **绝对禁止**：在 `forward` 中使用 Python 循环加索引赋值来构建需要梯度的输出。


# Q:方法二具体是怎么运算的？

为了让你彻底看懂这几步，我们**抛弃抽象的概念，拿一个具体的数字例子**，一步步拆解形状的变化和运算的物理含义。

假设：
- 输入 `X` 是一张单通道灰度图，形状为 `(1, 1, 3, 3)`（即 `B=1, C=1, H=3, W=3`）。
- 卷积核 `K` 形状为 `(2, 2)`（即 `h=2, w=2`）。

那么输出位置的个数 `N = (3-2+1) * (3-2+1) = 4`。我们期望输出一个 `(1, 1, 2, 2)` 的张量。

---

### 第 1 步：`F.unfold` —— 把滑动窗口“拍平”成列

**操作**：`patches = F.unfold(X, kernel_size=(2, 2))`

- **直观理解**：卷积核要在原图上滑动 4 次（左上、右上、左下、右下）。`unfold` 会把这 4 个 `2x2` 窗口全部提取出来，每个窗口拉伸成一个**列向量**。
- **形状变化**：
  - 输入形状：`(1, 1, 3, 3)`
  - 输出形状：`(B, C*h*w, N)` = `(1, 1*2*2, 4)` = **`(1, 4, 4)`**

**具体数据长什么样？**
假设原图 `X` 的数值为：
```
[[1, 2, 3],
 [4, 5, 6],
 [7, 8, 9]]
```
`unfold` 提取的 4 个窗口（按行优先排列）变成 4 列：
- **第1列**（位置(0,0)）：`[1, 2, 4, 5]^T`
- **第2列**（位置(0,1)）：`[2, 3, 5, 6]^T`
- **第3列**（位置(1,0)）：`[4, 5, 7, 8]^T`
- **第4列**（位置(1,1)）：`[5, 6, 8, 9]^T`

此时的 `patches` 矩阵（4行4列）：
```
列1  列2  列3  列4
[1,   2,   4,   5]   <- 窗口内的第1个元素（核位置(0,0)）
[2,   3,   5,   6]   <- 窗口内的第2个元素（核位置(0,1)）
[4,   5,   7,   8]   <- 窗口内的第3个元素（核位置(1,0)）
[5,   6,   8,   9]   <- 窗口内的第4个元素（核位置(1,1)）
```
> 每一列代表一个“感受野”窗口，每一行代表该窗口内同一个相对位置上的像素。

---

### 第 2 步：转置 `patches.transpose(1, 2)` —— 调换行列

**操作**：`patches.transpose(1, 2)`（维度顺序从 `(B, 特征, N)` 变为 `(B, N, 特征)`）

- **形状变化**：`(1, 4, 4)` → **`(1, 4, 4)`**（因为 B=1，前后都是4，形状数值没变，但**语义彻底变了**）。
- **为什么要转置？** 
  - 转置前：`(B, 4, 4)` 表示“4个特征行，4个样本列”。
  - 转置后：`(B, 4, 4)` 表示 **“4个样本（输出位置），每个样本有4个特征（窗口内的像素值）”**。
  
现在的矩阵（行是输出位置，列是窗口像素值）：
```
行1(位置0): [1, 2, 4, 5]
行2(位置1): [2, 3, 5, 6]
行3(位置2): [4, 5, 7, 8]
行4(位置3): [5, 6, 8, 9]
```

---

### 第 3 步：展平卷积核 `K.view(-1, 1)`

**操作**：`K_flat = K.view(-1, 1)`

- 原来的 `K` 是二维矩阵：
```
[[a, b],
 [c, d]]   (例如 a=0, b=1, c=2, d=3)
```
- `view(-1, 1)` 将其拉伸为 **列向量**，形状从 `(2, 2)` 变为 `(4, 1)`：
```
[a]   (即 [0])
[b]   (即 [1])
[c]   (即 [2])
[d]   (即 [3])
```
- **语义**：卷积核的 4 个权重值，变成了一个列向量，用来和上面的“特征行”做点积。

---

### 第 4 步：矩阵乘法 `@` —— 批量计算点积

**操作**：`out = patches.transpose(1, 2) @ K_flat`

- 左边矩阵形状：`(1, 4, 4)` —— 即 `(Batch, 行数=4, 列数=4)`
- 右边矩阵形状：`(4, 1)` —— 即 `(行数=4, 列数=1)`
- **矩阵乘法规则**：`(4, 4) @ (4, 1)` = 得到一个 `(4, 1)` 的结果。

**具体计算过程（就是内积）**：
- **输出位置1**（第一行）：`[1,2,4,5]` 与 `[a,b,c,d]` 点积 = `1*a + 2*b + 4*c + 5*d` = `1*0 + 2*1 + 4*2 + 5*3 = 0+2+8+15 = 25`
- **输出位置2**（第二行）：`[2,3,5,6]` 与核点积 = `2*0 + 3*1 + 5*2 + 6*3 = 0+3+10+18 = 31`
- **输出位置3**（第三行）：`[4,5,7,8]` 与核点积 = `4*0 + 5*1 + 7*2 + 8*3 = 0+5+14+24 = 43`
- **输出位置4**（第四行）：`[5,6,8,9]` 与核点积 = `5*0 + 6*1 + 8*2 + 9*3 = 0+6+16+27 = 49`

最终得到形状 `(B, N, 1)` = `(1, 4, 1)` 的数据，里面就是 `[25, 31, 43, 49]`。

---

### 最后一步（代码中省略但隐含）
代码最后通过 `.view(B, 1, H-h+1, W-w+1)` 将这 4 个数字重新排列成 `(1, 1, 2, 2)` 的输出特征图：
```
[[25, 31],
 [43, 49]]
```
这和你用双重循环算出来的结果完全一致（对照你原代码中的 `tensor([[19.,25.],[37.,43.]])`，仅数值不同，因为我用的是不同的 `X` 举例，但原理完全一样）。

---

### 一句话总结（精华版）
**`unfold` 把每个滑动窗口变成一列（Im2Col），转置后把这些列变成“样本行”，再把卷积核变成一列“权重向量”。此时，卷积运算就退化成了** 矩阵乘法（行与列的点积）**。这就是现代深度学习框架（如 cuDNN）底层加速卷积的核心数学原理（GEMM）。**


# Q:为什么原地赋值会破坏计算图？

这个问题问到了自动微分（Autograd）的底层机制核心。原地赋值（In-place assignment）之所以会破坏计算图，根本原因在于它**违背了链式求导法则对“历史记录”的依赖**。

为了把这件事说透，我们从“机制层”、“数据层”和“数学层”三个维度来拆解：

### 1. 机制层：计算图是“有向无环图”（DAG）
在 PyTorch 中，计算图本质上是一张记录数据流向的“施工蓝图”：

- **节点（Node）**：代表张量数据（如 `X`, `weight`, `Y`）。
- **边（Edge）**：代表运算操作（如 `mul`, `sum`, `add`）。
- **`grad_fn`**：每个非叶子节点的张量都带有 `grad_fn`，它指向“如何通过当前节点求导”的函数指针。

当你执行 `Y[i, j] = (X[i:i+h, j:j+w] * K).sum()` 时：
1. 右侧 `(X * K).sum()` 会生成一个**临时的中间张量**，这个张量带有 `grad_fn=<SumBackward>`。
2. 接着，你把**这个计算结果的值**取出来，**强行塞进**了预先用 `torch.zeros` 挖好的坑 `Y` 里。

**致命伤**：`Y` 这个节点的 `grad_fn` 原本是 `None`（因为是 `zeros` 创建的叶子节点）。赋值后，虽然 `Y` 里的数字变了，但它的 `grad_fn` **依然是 `None`**。计算图中那条从 `(X*K).sum()` 指向 `Y` 的“边”**凭空消失了**。反向传播时，引擎找不到 `Y` 是如何由 `X` 和 `K` 计算出来的路径，梯度自然就断了。

---

### 2. 数据层：PyTorch 的“版本计数器”（Version Counter）
PyTorch 为了确保反向传播的正确性，给每个参与求导的张量都加了一个**版本号（Version）**。

- 当你在 `forward` 中执行一个非原地操作（如 `c = a + b`）时，PyTorch 会记录下 `a` 和 `b` 在当前版本下的数值。
- 反向传播时（`backward`），引擎会回过头去取 `a` 和 `b` 在**前向传播那一刻**保存的数值来计算偏导数。

**原地赋值（`Y[i,j] = ...`）会直接递增该张量的版本计数器**。
更严重的是，如果被赋值的张量是一个**中间结果**（比如它参与了后续的加法 `+ self.bias`），那么当反向传播执行到 `+` 节点时，它需要用到前向时 `corr2d` 的输出值。但你的原地赋值把那个旧值覆盖了，版本号也对不上了。此时，PyTorch 会直接抛出经典的报错：

> `RuntimeError: one of the variables needed for gradient computation has been modified by an inplace operation`

这是 PyTorch 的“最后一道防线”，防止你用被篡改的数据做梯度计算。

---

### 3. 数学层：丢失了“自变量”与“因变量”的映射关系
从数学求导的角度看，反向传播依赖**雅可比矩阵（Jacobian）**的链式相乘。

假设我们只看其中一次循环的运算：
\( y = f(x, k) = \sum (x_{slice} \odot k) \)

在数学上，\( y \) 是 \( k \) 和 \( x \) 的函数。计算 \( \frac{\partial L}{\partial k} \) 必须用到 \( y \) 对 \( k \) 的表达式。

但当你做原地赋值 `Y[i,j] = 数值` 时，**这个赋值操作是 Python 层面的内存拷贝，不是张量运算（Tensor OP）**。PyTorch 的 Autograd 引擎只监听“张量运算”，不监听“内存地址拷贝”。因此，在自动微分引擎眼中，`Y` 和 `weight` 之间是**断开的**，没有任何数学关系。既然没有任何关系，`weight.grad` 就只能得到 `None` 或报错。

---

### 进一步思考：为什么直接 `+ bias` 没问题？
注意你的代码里有 `+ self.bias`，这个操作不会报错，因为 `+` 是标准的张量运算，它生成了一个**新的张量节点**，其 `grad_fn` 指向 `AddBackward`，链条是完整的。

---

### 结论（黄金法则）
**在 PyTorch 的 `forward` 函数中，凡是最终需要参与梯度回传的张量，都必须由已有的张量通过“张量运算（Tensor API）”组合产生，决不能通过“索引赋值（`[] =`）”或“内存拷贝”拼接产生。**

原地赋值只适合以下两种安全场景：
1. 不需要求导的中间变量（如模型推理时的数据归一化缓存）。
2. 在 `torch.no_grad()` 上下文中的参数更新（如优化器中的 `weight.data -= lr * grad`，这里的 `.data` 操作是刻意避开计算图的安全区）。
